RAPPORT DE TESTS - PROJET GESTION PLANNING

1. Introduction

L'objectif de ce travail est de valider le projet Gestion Planning par des tests appliques a l'architecture reelle du projet. Les tests ont ete realises sur le backend Spring Boot et sur le frontend Angular afin de couvrir la logique metier, les interactions entre couches et les principaux scenarios fonctionnels.

2. Structure reelle du projet

Backend:
- dossier: gestion-planing
- controllers: PlanningController, TaskController
- services: PlanningService, TaskService
- repositories: PlanningRepository, TaskRepository
- entities: Planning, Task
- enums: PlanningStatus, TaskStatus, TaskPriority
- gestion d'erreurs: GlobalExceptionHandler, BusinessException, ResourceNotFoundException
- DTO de reponse metrique:
  PlanningProgressResponse
  PlanningWeightedProgressResponse
  PlanningDailyLoadResponse
  PlanningEfficiencyResponse

Frontend:
- dossier: gestion-planning-front
- services Angular: planning.ts, task.ts, popup.service.ts
- pages/composants principaux:
  add-planning
  edit-planning
  list-planning
  planning-detail

4. Analyse des classes critiques a tester

PlanningService est critique car il contient la logique metier principale:
- validation des dates d'un planning
- recherche par mot-cle et statut
- calcul de progression simple
- calcul de progression ponderee
- calcul de charge journaliere
- calcul d'efficacite du planning

TaskService est critique car il gere:
- validation des horaires
- rattachement d'une tache a un planning
- recherche globale et par planning
- mise a jour et suppression de taches

Les controllers sont critiques car ils exposent les endpoints REST evalues:
- codes HTTP
- validation des requetes
- serialisation JSON
- gestion des erreurs 400 et 404

Les composants Angular sont critiques car ils portent les interactions utilisateur:
- chargement des donnees
- recherche
- creation
- modification
- suppression
- affichage des erreurs backend

5. Strategie de test retenue

Tests unitaires backend:
- ciblent PlanningService et TaskService
- utilisent Mockito pour isoler les repositories et dependances
- verifient la logique metier pure

Tests d'integration backend:
- utilisent @SpringBootTest et MockMvc
- executent les vrais controllers, services, repositories et handlers d'erreurs
- utilisent H2 pour eviter la dependance a MySQL pendant les tests

Tests frontend:
- services testes avec HttpTestingController
- composants testes avec TestBed et mocks de services
- verification des affichages et interactions utilisateur

9. Priorite sur la logique complexe

La priorite a ete donnee aux methodes suivantes:

- PlanningService.getPlanningProgress
  Couvre le calcul du pourcentage global a partir des statuts.
  Risque evite: afficher une progression incorrecte au tableau de bord.

- PlanningService.getPlanningWeightedProgress
  Couvre la ponderation par duree et priorite.
  Risque evite: metrique de performance trompeuse.

- PlanningService.getPlanningDailyLoad
  Couvre l'agregation des temps par jour.
  Risque evite: mauvaise estimation de la charge quotidienne.

- PlanningService.getPlanningEfficiency
  Couvre les ecarts entre taches, la densite, les penalites et le niveau final.
  Risque evite: score d'efficacite faux ou incoherent.

- TaskService.addTaskToPlanning / updateTask
  Couvre les validations horaires et l'interaction PlanningService -> TaskRepository.
  Risque evite: insertion de donnees invalides dans la base.

Backend dans IntelliJ:
1. Ouvrir le module gestion-planing
2. Lancer le dossier src/test/java ou une classe de test
3. Ou lancer la commande:
   .\mvnw.cmd test

Frontend dans IntelliJ ou terminal:
1. Ouvrir le module gestion-planning-front
2. Lancer:
   npm test -- --watch=false

14. Erreurs frequentes et corrections

Si MockMvc ne compile pas:
- verifier l'import adapte a Spring Boot 4:
  org.springframework.boot.webmvc.test.autoconfigure.AutoConfigureMockMvc

Si les tests d'integration essayent de se connecter a MySQL:
- verifier la presence de src/test/resources/application.properties
- verifier la dependance H2

Si les tests frontend echouent sur les processus:
- relancer dans un environnement autorisant le spawn des workers
- utiliser npm test -- --watch=false

Si les assertions Angular echouent apres mise a jour:
- verifier si le composant reset le formulaire apres succes
- comparer l'argument envoye au service avant le reset
